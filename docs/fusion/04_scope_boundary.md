# 04 范围边界

## 必须继承的工程风格约束（硬约束）

### 目标
定义 xy_quant 融合计划中必须继承的 SilverM 工程风格边界，避免在融合过程中偏离为“泛化方案”或“重造一套不兼容体系”。

### 规则
1. 必须保留统一数据接口思路：数据读取、指标计算、回测输入不能各自为政。
2. 必须保留明确的回测边界：回测引擎、数据适配、结果输出要分层，不混写业务逻辑。
3. 必须保留数据源可用性控制：健康检查、失败阈值、故障切换是上线前置条件。
4. 必须保留因子分析闭环：至少包含因子值、收益、统计、相关性或分位数检验。
5. 必须保留结果可追溯：每次运行应有 run_id、日志或数据库记录可复查。
6. 必须保留最小可观测输出：报告、图表、接口或页面至少一种可直接核验。

### 检查清单
- [ ] 是否存在把数据层、策略层、回测层混写在一个脚本里的情况。
- [ ] 是否存在没有健康检查就直接切主数据源的实现。
- [ ] 是否存在无法按 run_id 或日期复盘的回测结果。
- [ ] 是否存在因子分析只给结论不给统计过程的实现。
- [ ] 是否存在输出结果但无法落盘或无法审计的实现。

### 验收标准
- 任何新增实现都必须先通过这 6 条硬约束检查。
- 若某能力无法满足硬约束，必须标记为“不纳入融合范围”。
- 文档评审时可逐条检查，不允许使用“原则上”“建议上”替代硬约束表述。

## 范围说明
- 纳入：数据适配、因子分析、回测执行、报告输出、数据源治理。
- 暂不纳入：与 SilverM 风格无直接对应的复杂交易执行策略重构、过度前端重写、无验证闭环的实验性模块。
