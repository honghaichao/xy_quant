# 05 风险登记

## 风格偏离风险（新增分组）

### 目标
识别并约束 xy_quant 融合计划在吸收 SilverM 风格时可能出现的偏离，优先防止“看起来像融合、实际不可执行”的风险。

### 规则
- 风格偏离不是表述问题，而是落地缺失问题；只要无法验证，就视为偏离。
- 任何新增模块若不能映射到 SilverM 的证据文件或等价实现，就必须登记风险。
- 风险处理优先级高于功能扩展：先保证可验证，再谈增强。
- 所有缓解动作必须具体到文件、函数、检查点或验收动作。

### 检查清单
- [ ] 是否存在与 SilverM 证据不对应的新增术语或架构描述。
- [ ] 是否存在只写目标、不写检查方法的融合条目。
- [ ] 是否存在回测、数据源、因子分析三者之间断链的情况。
- [ ] 是否存在报告生成但无法复盘原始输入的情况。
- [ ] 是否存在“默认正确”但没有数据校验或日志支撑的实现。

### 验收标准
- 风格偏离风险必须在评审中单独列项，不得并入普通功能风险。
- 每个风险都要有明确触发条件、检测方式和缓解动作。
- 风险登记可直接指导代码审查、测试补齐与文档修订。

## 风格偏离风险清单

| 风险项 | 触发条件 | 影响 | 缓解动作 | 验证方式 |
|---|---|---|---|---|
| 风格空心化 | 只保留“对齐”“借鉴”但无落地模块 | 文档失真 | 强制补充路径、函数、表名 | 复查文档章节是否可核验 |
| 接口碎片化 | 数据层接口不统一 | 策略/回测复用失败 | 统一适配器与字段规范 | 检查适配器入口与返回格式 |
| 回测不可复盘 | 无 run_id 或结果落盘缺失 | 结果不可审计 | 强制记录 run_id、报告路径 | 检查 run_id 与报告文件 |
| 数据源失控 | 无健康检查或阈值治理 | 生产数据不稳定 | 引入健康检查/断路器 | 检查阈值与状态机 |
| 因子分析断链 | 只有信号没有统计验证 | 策略不可评估 | 补齐 IC、相关性、分位数收益 | 检查统计函数输出 |
